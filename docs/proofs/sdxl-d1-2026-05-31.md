# Proof — SDXL-Turbo D1 decode fix (rainbow → real image)

_Date: 2026-05-31. Mode: LFS-runtime (CUDA player build). Owner: forge lane._

## Problem
The forge ran on CUDA (`OnnxForge=True`, not placeholder) but decoded **near-noise → "rainbow"**
images (character-creation portrait, loading decorations, any generated icon).

## Root cause (hard evidence, static ONNX inspection)
`onnx.load` of the exported SDXL-Turbo encoders shows **every** CLIP layer is exposed as an output:
- `text_encoder` (ViT-L): `last_hidden_state`, `pooler_output`, `hidden_states.0 … hidden_states.12`
- `text_encoder_2` (ViT-bigG): `text_embeds`, `last_hidden_state`, `hidden_states.0 … hidden_states.32`

SDXL is trained on the **penultimate** CLIP hidden state (diffusers `hidden_states[-2]`):
- ViT-L → `hidden_states.11`, ViT-bigG → `hidden_states.31`.

`SdxlTurboPipeline` was reading **`last_hidden_state`** for both encoders. `last_hidden_state` is
post-final-layernorm (different scale/distribution), so the UNet cross-attention received
out-of-distribution conditioning → garbage ε → the single Euler step never denoised → noise decode.

Every other binding was already correct (UNet `sample/timestep/encoder_hidden_states/text_embeds/
time_ids` → `out_sample`; VAE `latent_sample` → `sample`), and the ε-step math
(`x0 = x − σ·ε`, `/0.13025`, decode, `(x+1)*0.5`) is correct. **The fix is the two output names.**

## Fix
`Assets/Scripts/Simulation/Forge/SdxlTurboPipeline.cs`:
`last_hidden_state` → `hidden_states.11` (enc1) and `hidden_states.31` (enc2). SD-1.5 correctly
stays on `last_hidden_state` (it is trained on the last hidden state). No other change.
